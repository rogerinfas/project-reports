\subsubsection{Semana 30/03/2026 -- Iteración/Sprint 1}

\textbf{Objetivo del Sprint:} establecer la base conceptual y de diseño de la propuesta, definiendo la estructura de arquitectura, el dominio de negocio y los primeros artefactos de análisis funcional.

\textbf{Actividades desarrolladas:}
\begin{itemize}
    \item Especificación del diseño: definición de estrategias bajo Clean Architecture.
    \item Modelado del dominio: identificación de Entidades, Value Objects y Agregados (Bus, Ruta, Paradero).
    \item Definición de reglas de negocio y lógica de dominio para el monitoreo.
    \item Redacción de historias de usuario iniciales en formato estándar (Como/Quiero/Para).
\end{itemize}

\textbf{Especificación del diseño (propuesta):}

La propuesta adopta \textbf{Clean Architecture} para desacoplar reglas de negocio de frameworks, base de datos e interfaces externas. La solución se organiza en cuatro capas:
\begin{itemize}
    \item \textbf{Dominio:} entidades, value objects, agregados y reglas de negocio puras.
    \item \textbf{Aplicación:} casos de uso, puertos de entrada/salida y coordinación de flujo.
    \item \textbf{Infraestructura:} persistencia, adaptadores externos, mensajería y geodatos.
    \item \textbf{Presentación:} API REST y cliente web para visualización en mapas.
\end{itemize}

Como estrategia de diseño, se aplican principios SOLID, inyección de dependencias e inversión de control. Esto permite que el núcleo del sistema (dominio) sea testeable y evolutivo sin dependencia directa de tecnologías de implementación.

\textbf{Modelado de dominio (entidades, value objects y agregados):}
\begin{itemize}
    \item \textbf{Agregado Ruta:} entidad raíz \texttt{Route}, asociada a su geometría \texttt{LineString} y a la secuencia de paraderos mediante \texttt{RouteStop}.
    \item \textbf{Agregado Bus:} entidad raíz \texttt{Bus}, con estado operativo y trazabilidad de asignación.
    \item \textbf{Agregado Paradero:} entidad \texttt{Stop}, con ubicación geoespacial \texttt{Point} y metadatos de referencia.
    \item \textbf{Value Objects:} \texttt{PlateNumber}, \texttt{RouteCode}, \texttt{GeoPoint}, \texttt{GeoLineString}, \texttt{BusStatus}, \texttt{DNI}, \texttt{Speed}, \texttt{Heading}.
\end{itemize}

\textbf{Reglas de negocio iniciales para monitoreo:}
\begin{enumerate}
    \item Un bus solo puede tener una asignación activa por turno.
    \item Una asignación requiere bus, conductor y ruta válidos.
    \item Solo buses en estado \texttt{activo} publican ubicación en tiempo real.
    \item Cada registro de ubicación debe incluir marca temporal válida.
    \item La placa del bus, el DNI del conductor y el código SIT de ruta son únicos.
    \item El cálculo de ETA usa historial de tráfico por segmento de ruta y condición de hora punta.
\end{enumerate}
